# Lowering reference

Zig++ has no hidden anything. Every construct lowers to plain Zig
text — `.zpp` source → `.zig` source — with no allocator inserted
behind your back, no implicit destructors, no exception machinery,
no implicit dispatch. This page is the side-by-side reference: for
each language construct, the source on one side and the exact Zig
the compiler emits on the other.

The lowering shown here is ground truth. The snapshots that back
this page live under `tests/lowering/snapshots/`; CI verifies the
compiler's output against them on every commit, so any drift between
this page and the actual compiler is a test failure.

## How to read this page

Every section presents a minimal `.zpp` snippet, the lowered `.zig`
the compiler emits, and a short paragraph on the rules driving the
transformation. Source comes either from a snapshot fixture or from
running `zpp lower` against an example in `examples/`. To inspect a
real file's lowering yourself:

```sh
zpp lower path/to/file.zpp
```

The output begins with `// Generated by zig++ frontend. DO NOT EDIT.`
and shows exactly what `zig` will see.

For a tour of running examples see `examples/` — every file there
demonstrates one or two constructs end-to-end. For the diagnostic
codes that the rules below cite (`Z0010`, `Z0021`, `Z0030`, …), see
the [Diagnostics](./diagnostics.md) reference.

## Construct index

| Construct                                                              | Lowers to                                                                  |
| ---------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| `trait T { ... }` + `impl T for S { ... }` + `fn f(x: impl T)`         | static dispatch via Zig `anytype` ([details](#trait--impl-trait-for-t))    |
| `fn f(x: dyn T)`                                                       | `zpp.Dyn(T_VTable)` fat pointer ([details](#dyn-trait))                    |
| `extern interface F { ... }`                                           | `extern struct F_ABI` with `callconv(.c)` ([details](#extern-interface))   |
| `using x = expr;`                                                      | `var x = expr; defer x.deinit();` ([details](#using))                      |
| `owned struct S`                                                       | plain `struct S` (sema-only check) ([details](#owned-struct))              |
| `own var x` + `move x`                                                 | sema-only checks (no runtime) ([details](#ownership-own-var-move-borrow))  |
| `requires(...)` / `ensures(...)`                                       | `zpp.contract.requires(...)` / `defer zpp.contract.ensures(...)` ([details](#contracts-requires--ensures)) |
| `effects(.X)` / `@effectsOf(f)`                                        | comment annotation; sema-checked ([details](#effects-and-effectsoff))      |
| `derive(.{ ... })`                                                     | injected comptime methods ([details](#derive))                             |

## `trait` + `impl Trait for T`

Traits are *sema-only* — the keyword does not produce a runtime
value of its own. From a single `trait` declaration the compiler
emits a vtable struct (used by `dyn` dispatch, see below) and a
`zpp.Dyn(...)` alias. From each `impl T for S` it emits the methods
*directly into S's body* (so `s.method()` resolves through Zig's
normal lookup) plus thunks that erase `*S` to `*anyopaque` for the
vtable. A `fn f(x: impl T)` parameter lowers to `fn f(x: anytype)`
and relies on Zig's monomorphization at every call site.

### Source `.zpp`

```zig
trait Greeter {
    fn greet(self) void;
}

const Hello = struct {
    name: []const u8,
};

impl Greeter for Hello {
    fn greet(self) void {
        std.debug.print("Hello, {s}!\n", .{self.name});
    }
}

fn welcome(who: impl Greeter) void {
    who.greet();
}
```

### Lowered `.zig`

```zig
const Greeter_VTable = struct {
    greet: *const fn (ptr: *anyopaque) void,
};
const Greeter = zpp.Dyn(Greeter_VTable);

const Hello = struct {
    name: []const u8,
    pub fn greet(self: *@This()) void {
        std.debug.print("Hello, {s}!\n", .{self.name});
    }
};

fn Greeter_greet_for_Hello(ptr: *anyopaque) void {
    const self: *Hello = @ptrCast(@alignCast(ptr));
    return self.greet();
}
pub const Greeter_impl_for_Hello: Greeter_VTable = .{
    .greet = Greeter_greet_for_Hello,
};

fn welcome(who: anytype) void {
    who.greet();
}
```

### Rules

- A `trait T { fn name(self) R; ... }` declaration emits
  `pub const T_VTable = struct { ... }` (one function pointer per
  method) and `pub const T = zpp.Dyn(T_VTable);` as a convenience
  alias.
- An `impl T for S { ... }` block does two things: injects each
  method as `pub fn` directly into the struct body, and synthesizes
  a flat free-function thunk plus a `T_impl_for_S: T_VTable` value
  for use with `dyn T`.
- A `fn f(x: impl T)` parameter is **static** — it lowers to
  `anytype` and Zig monomorphizes one copy per concrete type that
  satisfies `T`. No vtable, no indirection, direct method call.
- **Default methods** — when a trait method has a body, the
  compiler emits a `T_default_<name>` free function. Impls that
  omit the method get their vtable entry pointed at the default,
  so default implementations are inherited automatically.

See: `tests/lowering/snapshots/trait_simple.zig`,
`tests/lowering/snapshots/trait_default_method.zig`,
`examples/hello_trait.zpp`.

## `dyn Trait`

The visible-vtable form of dynamic dispatch. A `dyn T` value is a
fat pointer pairing an erased instance pointer with its vtable, and
every method call goes through `d.vtable.method(d.ptr, ...)` in the
lowered Zig — there is no implicit indirection.

### Source `.zpp`

```zig
trait Greeter {
    fn greet(self) []const u8;
}

fn shout(g: dyn Greeter) []const u8 {
    return g.greet();
}
```

### Lowered `.zig`

```zig
const Greeter_VTable = struct {
    greet: *const fn (ptr: *anyopaque) []const u8,
};
const Greeter = zpp.Dyn(Greeter_VTable);

fn shout(g: zpp.Dyn(Greeter_VTable)) []const u8 {
    return g.vtable.greet(g.ptr);
}
```

### Rules

- For each `trait T` the compiler synthesizes
  `pub const T_VTable = struct { ... }` and `pub const T = zpp.Dyn(T_VTable);`.
- For each `impl T for S` the compiler synthesizes
  `T_<method>_for_S(*anyopaque, ...)` thunks and
  `pub const T_impl_for_S: T_VTable = .{ ... }`.
- A parameter `fn f(x: dyn T)` lowers to
  `fn f(x: zpp.Dyn(T_VTable))`.
- A method call on a `dyn T` value lowers to
  `x.vtable.<method>(x.ptr, args...)` — explicit indirect call, no
  hidden dispatch.
- Boxing happens at the call site: typically
  `zpp.Dyn(T_VTable).init(S, &value, &T_impl_for_S)`, or the
  convenience `Dyn(...).from(...)`. See
  [Runtime → `zpp.Dyn(VTable)`](./runtime.md#zppdynvtable--visible-fat-pointer).

See: `tests/lowering/snapshots/dyn_call.zig`, `examples/dyn_plugin.zpp`,
`examples/event_bus.zpp`.

## `extern interface`

The C-ABI cousin of `dyn Trait`. Lowers to an `extern struct F_ABI`
with `callconv(.c)` function pointers, suitable for plugins loaded
via `dlopen`/`dlsym` or any other long-lived ABI boundary.

### Source `.zpp`

```zig
extern interface AudioPlugin {
    fn process(self, in: [*]const f32, out: [*]f32, n: usize) void;
}

const PassThrough = struct {};

impl AudioPlugin for PassThrough {
    fn process(self, in: [*]const f32, out: [*]f32, n: usize) void {
        _ = self;
        var i: usize = 0;
        while (i < n) : (i += 1) out[i] = in[i];
    }
}
```

### Lowered `.zig`

```zig
const AudioPlugin_ABI = extern struct {
    process: *const fn (ctx: *anyopaque, in: [*]const f32, out: [*]f32, n: usize) callconv(.c) void,
};
const AudioPlugin_VTable = AudioPlugin_ABI;

fn AudioPlugin_process_for_PassThrough(
    ptr: *anyopaque,
    in: [*]const f32, out: [*]f32, n: usize,
) callconv(.c) void {
    const self: *PassThrough = @ptrCast(@alignCast(ptr));
    _ = self;
    var i: usize = 0;
    while (i < n) : (i += 1) out[i] = in[i];
}

pub const AudioPlugin_impl_for_PassThrough: AudioPlugin_VTable = .{
    .process = AudioPlugin_process_for_PassThrough,
};
```

### Rules

- `extern interface F { fn name(...) R; ... }` lowers to an
  `extern struct F_ABI` whose fields are `*const fn (...) callconv(.c) R`.
- The compiler injects `ctx: *anyopaque` as the first parameter of
  every function pointer (the erased `self`).
- Field order is positional (extern struct ABI guarantee), so the
  vtable layout is frozen — older plugin binaries can call newer
  hosts and vice versa as long as fields are appended.
- The `impl` thunks use the same `callconv(.c)` so they're directly
  storable in the `_ABI` struct.
- `extern interface` and `dyn Trait` are **not** interchangeable:
  `dyn` uses Zig calling convention with no ABI guarantees; `extern
  interface` locks in the C convention for stable FFI.

See: `examples/extern_plugin.zpp`.

## `using`

Explicit RAII binder. Lowers literally to `var x = expr;` plus a
`defer x.deinit();`. There is no runtime helper called — this is a
pure compiler-side rewrite.

### Source `.zpp`

```zig
fn run(allocator: std.mem.Allocator) !void {
    using arena = zpp.ArenaScope.init(allocator);
    const a = arena.allocator();
    const buf = try a.alloc(u8, 16);
    _ = buf;
}
```

### Lowered `.zig`

```zig
fn run(allocator: std.mem.Allocator) !void {
    var arena = zpp.ArenaScope.init(allocator);
    defer arena.deinit();
    const a = arena.allocator();
    const buf = try a.alloc(u8, 16);
    _ = buf;
}
```

### Rules

- `using x = expr;` lowers to:
  ```zig
  var x = expr;
  defer x.deinit();
  ```
- Sema verifies that `@TypeOf(x)` has
  `pub fn deinit(self: *@This()) void`. If missing, **Z0011** fires
  — without a `deinit` the generated `defer` would not type-check.
- The `defer` runs on every scope exit (normal return, error
  return, break, panic if defers run on the panic path).
- `using` is a *statement binder*, not an expression — it can only
  appear at statement position.

See: `tests/lowering/snapshots/hello_using.zig`,
`examples/owned_file.zpp`.

## `owned struct`

A `owned struct S` is a *sema-checked promise*. The struct itself
lowers to a plain Zig `struct` with no extra runtime tag — but sema
enforces that it must declare a `deinit` method, and the canonical
release pattern is to pair it with `using`.

### Source `.zpp`

```zig
owned struct LogFile {
    file: std.fs.File,

    pub fn init(path: []const u8) !LogFile {
        return .{ .file = try std.fs.cwd().createFile(path, .{ .truncate = true }) };
    }

    pub fn deinit(self: *LogFile) void {
        self.file.close();
    }
}

pub fn main() !void {
    using log = try LogFile.init("trace.log");
    _ = log;
}
```

### Lowered `.zig`

```zig
const LogFile = struct {
    file: std.fs.File,

    pub fn init(path: []const u8) !LogFile {
        return .{ .file = try std.fs.cwd().createFile(path, .{ .truncate = true }) };
    }

    pub fn deinit(self: *LogFile) void {
        self.file.close();
    }
};

pub fn main() !void {
    var log = try LogFile.init("trace.log");
    defer log.deinit();
    _ = log;
}
```

### Rules

- The `owned` modifier is *erased* at codegen time. The struct
  becomes a plain `pub const S = struct { ... };` — no extra fields,
  no runtime tag.
- Sema requires `pub fn deinit(self: *@This()) void`. Missing →
  **Z0010**.
- The compiler does **not** auto-call `deinit` for you — you still
  need `using` (which emits the `defer ... .deinit();`) or a manual
  `defer x.deinit();`.
- The canonical pattern is `owned struct` + `using` together.

See: `examples/owned_file.zpp`.

## Ownership: `own var`, `move`, borrow

The affine ownership axis is *boring at runtime*. `own var x = ...`
lowers to plain `var x = ...`; `move x` lowers to just `x`;
statement-form `move x;` becomes `// move x` after sema runs. All
the work is sema-side: a multi-borrow + block-scope-aware borrow
checker fires **Z0020** (use-after-move) and **Z0021**
(borrow-invalidated-by-move) when the rules are violated.

### Source `.zpp`

```zig
own var seven = Token{ .id = 0 };
seven.id = 7;
consume(move seven);                 // OK: no borrow on `seven`

own var scoped = Token{ .id = 0 };
scoped.id = 9;
{
    const ref = &scoped;             // borrow scoped to inner block
    std.debug.print("peek id={d}\n", .{ref.id});
}
consume(move scoped);                // OK: borrow already retired

// own var doomed = Token{ .id = 9 };
// const ref = &doomed;              // borrow lives in this scope
// std.debug.print("ref id={d}\n", .{ref.id});
// consume(move doomed);             // Z0021: cannot move while borrowed
```

### Lowered `.zig`

```zig
var seven = Token{ .id = 0 };
seven.id = 7;
consume(seven);

var scoped = Token{ .id = 0 };
scoped.id = 9;
{
    const ref = &scoped;
    std.debug.print("peek id={d}\n", .{ref.id});
}
consume(scoped);
```

### Rules

- `own var x = expr;` lowers to `var x = expr;`. The `own` keyword
  is a sema-only tag.
- `move x` (expression form) lowers to `x`. Statement-form
  `move x;` becomes `// move x` after sema.
- **Z0020 use-after-move** — reading `x` after `move x` in the same
  scope is rejected.
- **Z0021 borrow-invalidated-by-move** — taking `&x` (or
  `&x.field`) at lexical depth `D` and then `move x` while the
  borrow is still live (block depth has not yet dropped below `D`)
  is rejected.
- The borrow checker is **multi-borrow + block-scope-aware**:
  simultaneous shared borrows are fine; closing a block retires
  borrows it contained; any `move` with a live borrow fires Z0021.

See: `examples/borrow_check.zpp`.

## Contracts: `requires` / `ensures`

Runtime contracts compile to `zpp.contract.requires(...)` for
preconditions and `defer zpp.contract.ensures(...)` for
postconditions. Both compile to *zero* in `ReleaseFast` /
`ReleaseSmall` via a comptime guard inside `zpp.contract` — see
[Runtime → `zpp.contract`](./runtime.md#zppcontract--runtime-contracts).

### Source `.zpp`

```zig
fn binarySearch(comptime T: type, xs: []const T, target: T) ?usize
    where T: Ord
    requires(isSorted(T, xs))
{
    var lo: usize = 0;
    var hi: usize = xs.len;
    while (lo < hi) {
        const mid = lo + (hi - lo) / 2;
        const c = xs[mid].cmp(target);
        if (c == 0) return mid;
        if (c < 0) lo = mid + 1 else hi = mid;
    }
    return null;
}
```

### Lowered `.zig`

```zig
fn binarySearch(comptime T: type, xs: []const T, target: T) ?usize {
    zpp.contract.requires(isSorted(T, xs), "requires: isSorted(T, xs)" ++ " in binarySearch");
    var lo: usize = 0;
    var hi: usize = xs.len;
    while (lo < hi) {
        const mid = lo + (hi - lo) / 2;
        const c = xs[mid].cmp(target);
        if (c == 0) return mid;
        if (c < 0) lo = mid + 1 else hi = mid;
    }
    return null;
}
```

### Rules

- `requires(cond)` lowers to
  `zpp.contract.requires(cond, "requires: <textual cond> in <fn_name>");`
  as the first statement of the body. The textual condition is
  captured verbatim at parse time and echoed in the panic message.
- `ensures(cond)` lowers to
  `defer zpp.contract.ensures(cond, "ensures: <textual cond> in <fn_name>");`
  so the condition is evaluated at every scope exit (final state).
- `where T: Trait` is informational at lowering — it documents the
  constraint and informs the trait satisfiability check, but the
  clause itself does not survive into the lowered Zig.
- In `ReleaseFast` and `ReleaseSmall`, both `requires` and
  `ensures` inline to nothing via the `checks_on` comptime guard —
  the message string and the condition evaluation are statically
  eliminated.

See: `examples/contracts_sort.zpp`.

## `effects(...)` and `@effectsOf(f)`

Effects are sema-only. The lowered Zig for an `effects(.noalloc) fn`
is just the function body with a comment annotation — there is no
runtime check, no struct field, no `@noOptimize` hint. `@effectsOf(f)`
lowers to a comptime `[]const u8` literal listing whatever sema
inferred for `f`.

### Source `.zpp`

```zig
fn pure(x: u32) u32 {
    return x *% 31 +% 0xdeadbeef;
}

fn noisy(a: std.mem.Allocator) !void {
    const buf = try a.alloc(u8, 8);
    defer a.free(buf);
    std.debug.print("noisy: allocated {d} bytes\n", .{buf.len});
}

effects(.custom("net")) fn networking() void {}

pub fn main() !void {
    const pure_effects:  []const u8 = @effectsOf(pure);
    const noisy_effects: []const u8 = @effectsOf(noisy);
    const net_effects:   []const u8 = @effectsOf(networking);
}
```

### Lowered `.zig`

```zig
fn pure(x: u32) u32 {
    return x *% 31 +% 0xdeadbeef;
}

fn noisy(a: std.mem.Allocator) !void {
    const buf = try a.alloc(u8, 8);
    defer a.free(buf);
    std.debug.print("noisy: allocated {d} bytes\n", .{buf.len});
}

// effects: custom("net")
fn networking() void {}

pub fn main() !void {
    const pure_effects:  []const u8 = "";
    const noisy_effects: []const u8 = "alloc,io";
    const net_effects:   []const u8 = "custom(\"net\")";
}
```

### Rules

- `effects(.X)` annotations lower to a `// effects: <list>` comment
  directly above the fn declaration. There is no runtime
  representation.
- Sema runs bottom-up effect inference and compares the inferred
  set against the declaration. Declaring `.noalloc` while the
  inferred set contains `.alloc` fires **Z0030** — same shape for
  `.noio`, `.nopanic`, `.noasync`.
- `.custom("X")` and `.nocustom("X")` are user-defined tags. They
  propagate one round through same-file callees. A
  `.nocustom("X")`-vs-`.custom("X")` mismatch fires **Z0060**.
- `@effectsOf(f)` lowers to a comptime string literal. The format
  is `"alloc,io,panic,async"` (in that fixed order), followed by
  any `custom("X")` entries (with the inner quotes escaped). Pure
  functions produce `""`.
- If `f` isn't a same-file fn name, lowering still emits `""` so
  the surrounding code keeps compiling, but sema reports **Z0050**
  so you know the queryable result is meaningless. Cross-file
  inference is out of MVP scope.

See [Diagnostics → Effects](./diagnostics.md#effects) for the codes
this section cites.

## `derive(.{ ... })`

`derive` injects comptime helpers from
[`zpp.derive`](./runtime.md#zppderive--comptime-trait-synthesis)
directly into the struct body. After lowering, the methods resolve
through Zig's normal struct method lookup — there is no extension
trait or vtable.

### Source `.zpp`

```zig
struct User {
    id: u64,
    name: []const u8,
} derive(.{ Hash, Eq, Ord, Default, Compare });
```

### Lowered `.zig`

```zig
const User = struct {
    id: u64,
    name: []const u8,

    pub fn hash(self: @This()) u64 {
        return zpp.derive.Hash(@This()).hash(self);
    }
    pub fn eq(self: @This(), other: @This()) bool {
        return zpp.derive.Eq(@This()).eq(self, other);
    }
    pub fn cmp(self: @This(), other: @This()) i32 {
        return zpp.derive.Ord(@This()).cmp(self, other);
    }
    pub fn default() @This() {
        return zpp.derive.Default(@This()).default();
    }
    pub fn lt(self: @This(), other: @This()) bool {
        return zpp.derive.Compare(@This()).lt(self, other);
    }
    pub fn le(self: @This(), other: @This()) bool {
        return zpp.derive.Compare(@This()).le(self, other);
    }
    pub fn gt(self: @This(), other: @This()) bool {
        return zpp.derive.Compare(@This()).gt(self, other);
    }
    pub fn ge(self: @This(), other: @This()) bool {
        return zpp.derive.Compare(@This()).ge(self, other);
    }
};
```

### Rules

- `} derive(.{ Trait1, Trait2, ... });` after a struct decl expands
  to a series of `pub fn` (and occasionally `pub const`) injections
  inside the struct body, each delegating to the matching
  `zpp.derive.<Trait>(@This())` helper.
- Multiple derives can be listed in a single clause — order in the
  list maps 1:1 to the order of the lowered `pub fn` declarations.
- Allocator-requiring derives (`Clone`, `Json`, `Serialize`,
  `FromStr`) get an `allocator: std.mem.Allocator` parameter on the
  injected method automatically; you call them as
  `a.serialize(allocator)` and never spell the allocator inside the
  struct.
- `Debug` and the compiler-level `Json` derive emit
  `pub const debug = zpp.derive.Debug(@This());` style aliases
  alongside the methods, so `User.debug.format(a, w)` and
  `User.json.parse(s, arena)` work.

See: `tests/lowering/snapshots/derive_extras.zig`,
`examples/derive_user.zpp`. For the full per-derive method table
and constraints, see
[Runtime → `zpp.derive`](./runtime.md#zppderive--comptime-trait-synthesis).
