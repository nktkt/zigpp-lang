# CLI reference

`zpp` is the single front door to the Zig++ toolchain. It is a thin
driver: `argv[1]` selects a subcommand and the driver dispatches to a
handler module under `tools/` (`zpp_fmt.zig`, `zpp_lsp.zig`,
`zpp_doc.zig`, `zpp_migrate.zig`, …). Every subcommand operates on
`.zpp` source and either lowers it to `.zig` for execution, runs a
parse + sema pass, or hands the AST off to a downstream tool. There
is no hidden global state — each invocation is one shot.

The CLIs install under `zig-out/bin/` after `zig build`. The main
driver is `zpp`; four sibling binaries (`zpp-fmt`, `zpp-lsp`,
`zpp-doc`, `zpp-migrate`) ship the same logic as standalone
executables for editors and build pipelines that want to skip the
dispatcher. For diagnostic codes referenced by `zpp check` and
`zpp explain`, see the [Diagnostics](./diagnostics.md) page.

## At a glance

| Subcommand                       | Summary                                                                          |
| -------------------------------- | -------------------------------------------------------------------------------- |
| [`build [path]`](#zpp-build)     | Lower `.zpp` → `.zig` under `.zpp-cache/` and run `zig build`.                   |
| [`run <file.zpp>`](#zpp-run)     | Lower one source file and execute it via `zig run`.                              |
| [`lower <file.zpp>`](#zpp-lower) | Print lowered `.zig` to stdout.                                                  |
| [`fmt [paths...]`](#zpp-fmt)     | Format `.zpp` files in place.                                                    |
| [`check [paths...]`](#zpp-check) | Parse + sema; exit nonzero on diagnostic.                                        |
| [`watch [paths...]`](#zpp-watch) | Re-run `check` whenever any `.zpp` under `<paths>` changes.                      |
| [`doc [paths...]`](#zpp-doc)     | Generate Markdown (default) or HTML (`--html`) docs.                             |
| [`migrate <file.zig>`](#zpp-migrate) | Suggest `.zpp` rewrites for an existing `.zig` file.                         |
| [`lsp`](#zpp-lsp)                | Start the LSP server on stdin/stdout.                                            |
| [`init <name>`](#zpp-init)       | Scaffold a new project (`--template exe \| lib \| plugin`).                      |
| [`test [paths...]`](#zpp-test)   | Lower `.zpp` files and run their `test` blocks via `zig test`.                   |
| [`explain <Z####>`](#zpp-explain) | Long-form explanation of a diagnostic code.                                      |
| [`version`](#zpp-version)        | Print compiler version.                                                          |
| [`help [subcommand]`](#zpp-help) | Show top-level usage or details for one subcommand.                              |

## Sibling binaries

`zpp-fmt`, `zpp-lsp`, `zpp-doc`, and `zpp-migrate` are the same logic
exposed as standalone binaries so editors and build pipelines can
shell out without spawning the full driver. Each one shares the
underlying `.zig` module with the corresponding `zpp` subcommand, so
behaviour and flags stay in lockstep — there is no second
implementation to drift.

| Binary       | Subcommand equivalent | Notes                                                                              |
| ------------ | --------------------- | ---------------------------------------------------------------------------------- |
| `zpp-fmt`    | `zpp fmt`             | Drop-in for editor "format on save" hooks; no driver overhead.                     |
| `zpp-lsp`    | `zpp lsp`             | Speaks LSP over stdio. VS Code, Neovim, Emacs, Helix all point at this directly.   |
| `zpp-doc`    | `zpp doc`             | Standalone Markdown/HTML doc generator; convenient for static-site pipelines.      |
| `zpp-migrate`| `zpp migrate`         | One-shot `.zig` → `.zpp` rewriter; pipes cleanly into `diff`/`patch`.              |

When in doubt, prefer the `zpp <subcommand>` form — the sibling
binaries exist for tooling that already knows which capability it
needs.

## A note on `--help`

Use `zpp help <subcommand>`. The driver does not implement a
per-subcommand `--help` parser, so `zpp <cmd> --help` is inconsistent
across subcommands: `zpp explain --help` is rejected, `zpp fmt --help`
is treated as a path, and so on. `zpp help <cmd>` is the only reliable
form.

## `zpp build`

Lowers every `.zpp` under `[path]` (default `.`) into `.zpp-cache/`,
mirroring the directory layout, then invokes `zig build`. If a
top-level `build.zpp` exists at the project root it is lowered to
`build.zig` first (with a `// Generated by zpp from build.zpp.`
header) so the Zig toolchain can drive the build normally; an
existing hand-written `build.zig` always takes precedence and is
never overwritten. When no `build.zig` exists at all, `zpp` writes a
minimal shim at `.zpp-cache/build.zig` that wires the lowered entry
point plus the `zpp` runtime module.

**Usage**

```sh
zpp build [path] [-- <args forwarded to `zig build`>]
```

**Behavior**

- Walks `[path]` recursively, lowering each `*.zpp` to `.zpp-cache/*.zig`.
- mtime-based re-lowering for `build.zpp`; warns once and ignores
  `build.zpp` if a hand-written `build.zig` is present.
- `--` forwards everything after it verbatim to `zig build`.

**Env vars**

- `ZPP_LIB` — overrides the location of `lib/zpp.zig` runtime when
  emitting the auto-shim (falls back to `lib/zpp.zig`,
  `../lib/zpp.zig`, an upward walk, then paths relative to the
  `zpp` executable).

**Example**

```sh
zpp build                       # lower everything + run `zig build`
zpp build -- run                # forward `run` step to zig build
zpp build src -- test --release=fast
```

## `zpp run`

Lowers a single `.zpp` source to Zig, writes it into the per-project
`./.zpp-cache/`, then invokes `zig run` on the lowered file with the
`zpp` runtime wired in as a module.

**Usage**

```sh
zpp run <file.zpp> [extra args forwarded to `zig run`]
```

There is no `--` separator parser — every argument after `<file.zpp>`
is appended verbatim to the `zig run` argv.

**Cache**

`./.zpp-cache/<stem>.zig` (e.g. `hello_trait.zpp` →
`.zpp-cache/hello_trait.zig`). Overwritten on every invocation.

**Example**

```text
$ zpp run examples/hello_trait.zpp
Hello, Ada!
```

## `zpp lower`

Prints the lowered Zig source for exactly one `.zpp` file to stdout —
a pure pipe, no filesystem side effects, no `zig` invocation.

**Usage**

```sh
zpp lower <file.zpp>
```

Requires exactly one positional argument. Output begins with the
banner `// Generated by zig++ frontend. DO NOT EDIT.`. There is no
`-o` flag — redirect stdout instead:

```sh
zpp lower foo.zpp > foo.zig
```

## `zpp fmt`

Re-emits `.zpp` source with canonical whitespace: 4-space indentation,
collapsed runs of blank lines, trimmed trailing whitespace,
keyword-aware spacing for the Zig++ vocabulary, and a guaranteed
trailing newline.

**Usage**

```sh
zpp fmt [--check] [paths...]
```

Each path may be a `.zpp` file or a directory; directories are walked
recursively. With no paths, the current directory is used.

**Behavior**

- Rewrites files **in place** only when the formatted bytes differ.
- Always prints `zpp fmt: processed N, changed N` to stdout.
- `--check` doesn't write; prints `'<path>' would be reformatted` to
  stderr and exits non-zero if any file drifts. Useful in CI.

The standalone `zpp-fmt` is functionally identical (same lexer,
formatter, in-place writes); the only differences are that it accepts
`--help` and skips the summary line.

## `zpp check`

Parses and semantically analyses `.zpp` files and prints diagnostics.
No codegen, no lowering, no IR.

**Usage**

```sh
zpp check [paths...]
```

Path semantics are the same as `fmt` — files or directories,
recursive walk, default to `.`.

**Exit codes**

- `0` — no error-severity diagnostics across all files.
- `1` — at least one file produced an error diagnostic, or failed
  to read or stat. Per-file failures are reported but do not abort
  the run; every path is still visited.

See the [Diagnostics](./diagnostics.md) page for the full code
catalog. `zpp explain Z####` resolves any code printed by `check`.

## `zpp watch`

Re-runs `zpp check` over the given paths whenever any tracked `.zpp`
file changes. Long-running; exit with Ctrl-C.

**Usage**

```sh
zpp watch [paths...]
```

**Behavior**

- mtime-based polling, ~500ms tick. No inotify / FSEvents / kqueue.
- Any of: file count change, missing tracked path, new path, or any
  mtime delta triggers a re-run.
- Status lines (`watching N path(s)`, `ok` / `errors` after each
  re-run) on stdout; diagnostics on stderr.
- Multiple writes within one tick collapse into one re-run.

## `zpp doc`

Generates API reference documents from `///` doc comments attached to
top-level `trait`, `fn`, `struct`, `owned struct`, and
`extern interface` declarations.

**Usage**

```sh
zpp doc [paths...] [-o <dir>] [--html]
```

**Flags**

- `-o, --out <dir>` — output directory (default `docs/`, created if missing).
- `--markdown`, `--md` — Markdown output (default).
- `--html` — self-contained HTML with embedded CSS, table of
  contents, anchor IDs, and a generated `index.html`.

Output mirrors the input's relative path:
`examples/hello_trait.zpp` → `docs/hello_trait.md` (or `.html`).

**Example**

```text
$ zpp doc examples/hello_trait.zpp
zpp doc: wrote 1 markdown file(s) under docs/
```

The standalone `zpp-doc` accepts the same arguments and is intended
for build pipelines that don't want to ship the full `zpp` driver.

## `zpp migrate`

Static analyser that scans a Zig source file for idioms with a Zig++
equivalent and emits unified-diff-style suggestions. With `--write`
it edits the file in place, applying only the rewrites it has marked
auto-safe.

**Usage**

```sh
zpp migrate <file.zig> [--write]
```

**Rewrite patterns**

| #   | Zig pattern                                              | Zig++ rewrite                              | `--write`      |
| --- | -------------------------------------------------------- | ------------------------------------------ | -------------- |
| 1   | `var x = try X.init(...);` + later `defer x.deinit();`   | `using x = try X.init(...);`               | auto           |
| 2   | `fn f(comptime T: type, x: *T, ...)`                     | `impl SomeTrait` parameter                 | suggestion     |
| 3   | `struct { ... pub fn init/deinit ... }`                  | `owned struct`                             | suggestion     |
| 4   | `if (cond) @panic("msg");`                               | `zpp.contract.requires(cond, "msg");`      | auto if `void` |
| 5   | manual `pub fn hash`                                     | `derive(.{ .Hash })`                       | suggestion     |
| 6   | struct with allocator-freeing `deinit`                   | `owned struct`                             | auto           |
| 7   | error-returning fn with no I/O markers                   | `effects(.noio)`                           | suggestion     |
| 8   | `if (!cond) return error.X;` near a precondition comment | `zpp.contract.requires(cond, "...");`      | suggestion     |

The standalone `zpp-migrate` is the same code path; both forms
accept `<file.zig> [--write]`.

## `zpp lsp`

A Language Server Protocol implementation for Zig++ that speaks LSP
over stdio. Built on the same parse/sema pipeline as `zpp check`.

**Usage** — typically launched by an editor; not invoked by humans.
There is no `--help`, no flags, no TCP transport.

**Capabilities** (verified against `tools/zpp_lsp.zig`)

- `textDocument` sync (full), formatting, hover, document & workspace
  symbols, definition / typeDefinition / implementation, references,
  rename (with `prepareProvider`), completion (trigger characters
  `.`, `:`), signatureHelp (trigger characters `(`, `,`),
  diagnostics (push + pull), code actions (`quickfix` — every
  diagnostic gets `Explain Z####`; `Z0010` and `Z0040` ship
  `WorkspaceEdit` auto-fixes), code lens (effect / derive / owned),
  call hierarchy, semantic tokens (`full`, `full/delta`, `range`),
  inlay hints, folding ranges.

**Editor configuration (generic LSP client)**

```json
{
  "command": "zpp-lsp",
  "args": [],
  "filetypes": ["zigpp"],
  "rootPatterns": ["build.zpp", "build.zig", ".git"],
  "transport": "stdio"
}
```

The `zpp lsp` invocation is identical and useful when only `zpp` is
on `PATH`.

## `zpp init`

Scaffolds a new Zig++ project under `<name>/`. Refuses to overwrite
an existing directory. The project name must match
`[A-Za-z][A-Za-z0-9_-]*`.

**Usage**

```sh
zpp init <name> [--template exe|lib|plugin]
zpp init --list                    # list every template
```

**Templates**

| Template | Description                                                        |
| -------- | ------------------------------------------------------------------ |
| `exe`    | Minimal executable: `src/main.zpp` + `build.zig`. **(default)**    |
| `lib`    | Library: public-API surface in `src/<name>.zpp` + tests, no main.  |
| `plugin` | Plugin: `extern interface` + host + plugin shared lib (C-ABI).     |

**Files generated (`exe` template)**

| Path             | Purpose                                                              |
| ---------------- | -------------------------------------------------------------------- |
| `build.zig`      | Wires `zpp lower src/main.zpp` and the `zpp` runtime module.         |
| `build.zig.zon`  | Manifest with `minimum_zig_version = "0.15.0"` and `zigpp` dep.      |
| `src/main.zpp`   | Starter program demonstrating `trait` + `impl`.                      |
| `.gitignore`     | Ignores `zig-out/`, `.zig-cache/`, `.zpp-cache/`, `.DS_Store`.       |
| `README.md`      | Quickstart with `zig fetch --save` + `zig build run`.                |

**Next steps**

```sh
cd <name>
zig fetch --save git+https://github.com/nktkt/zigpp-lang
zig build run
```

## `zpp test`

Lowers every `.zpp` under `[paths]` (default `.`) into
`.zpp-cache/<rel>.zig`, then runs `zig test` against each lowered
file. Exits non-zero if any file fails.

**Usage**

```sh
zpp test [paths...] [--filter <p>] [--release] [-v]
```

**Flags**

- `--filter <p>` — passed through to `zig test --test-filter`.
- `--release` — build tests in release mode.
- `-v` — verbose: print every command being executed.

## `zpp explain`

Print a long-form explanation of a Zig++ diagnostic code, or
`--list` to print every code with a one-line summary. See the
[Diagnostics](./diagnostics.md) reference page for the full catalog.

**Usage**

```sh
zpp explain <Z####>            # human-readable explanation
zpp explain <Z####> --json     # JSON for IDE consumption
zpp explain --list             # one-line summary of every code
zpp explain --list --json      # same, as JSON
```

**JSON shape (single code)**

```json
{
  "code": "Z0010",
  "title": "owned struct missing deinit",
  "summary": "...",
  "explain": "Z0010: owned struct missing deinit\n\n...",
  "examples": [
    { "kind": "trigger", "snippet": "..." },
    { "kind": "fix",     "snippet": "..." }
  ]
}
```

## `zpp version`

```text
$ zpp version
zpp 0.1.0 (Zig++ research compiler)
```

Also reachable as `zpp --version` or `zpp -V`.

## `zpp help`

Show top-level usage, or — when a subcommand is supplied — print
that subcommand's help block.

**Usage**

```sh
zpp help                       # top-level usage
zpp help <subcommand>          # detail for one subcommand
```

Reachable as `zpp help`, `zpp --help`, `zpp -h`, or `zpp` with no
args (the latter exits with usage-error). Always prefer
`zpp help <subcommand>` over `zpp <subcommand> --help`; see the
note at the top of this page.
