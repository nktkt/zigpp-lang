# Tutorial — your first 10 minutes with Zig++

This walks you from "I just heard about Zig++" to a running program
that uses traits, RAII, and a runtime contract. By the end you will
have a project on disk, modified two ways, run twice. Total time
~10 minutes.

## Prerequisites

- **Zig 0.15 or later** on your `PATH`.
  ```sh
  zig version
  # 0.15.2 (or newer)
  ```
  If you don't have it, grab a release from
  [ziglang.org/download](https://ziglang.org/download/).

- A POSIX shell or PowerShell. The CI matrix tests both.

## Step 1 — get the `zpp` CLI

Clone and build once. The binaries land in `zig-out/bin/`.

```sh
git clone https://github.com/nktkt/zigpp-lang
cd zigpp-lang
zig build
```

You should see `zig-out/bin/zpp` and 4 sibling tools (`zpp-fmt`,
`zpp-lsp`, `zpp-doc`, `zpp-migrate`). Add the directory to your
`PATH` for convenience:

```sh
export PATH="$PWD/zig-out/bin:$PATH"
zpp version
# zpp 0.1.0 (Zig++ research compiler)
```

## Step 2 — scaffold a new project

```sh
cd ..
zpp init hello-zpp
cd hello-zpp
```

You now have:

```
hello-zpp/
  build.zig            invokes zpp lower at build time
  build.zig.zon        depends on the zigpp package
  src/
    main.zpp          a tiny trait + impl example
  .gitignore
  README.md
```

Open `src/main.zpp` in your editor:

```zig
const std = @import("std");
const zpp = @import("zpp");

trait Greeter {
    fn greet(self) void;
}

const Friendly = struct {
    name: []const u8,
};

impl Greeter for Friendly {
    fn greet(self) void {
        std.debug.print("hello from zpp, {s}!\n", .{self.name});
    }
}

fn welcome(who: impl Greeter) void {
    who.greet();
}

pub fn main() !void {
    var f = Friendly{ .name = "world" };
    welcome(&f);
}
```

## Step 3 — pin the dependency and run

The first build needs to record the `zigpp` package hash. Zig will
prompt you with the value to paste; `--save` does this automatically:

```sh
zig fetch --save git+https://github.com/nktkt/zigpp-lang
```

Then build and run:

```sh
zig build run
# hello from zpp, world!
```

If it ran, the whole pipeline worked: `.zpp` source → lowered `.zig`
→ Zig compiler → executable.

## Step 4 — see the lowered Zig

Curious what the compiler emitted?

```sh
zpp lower src/main.zpp
```

Output (abbreviated):

```zig
// Generated by zig++ frontend. DO NOT EDIT.

const std = @import("std");
const zpp = @import("zpp");

const Greeter_VTable = struct {
    greet: *const fn (ptr: *anyopaque) void,
};
const Greeter = zpp.Dyn(Greeter_VTable);

const Friendly = struct {
    name: []const u8,
    pub fn greet(self: *@This()) void {
        std.debug.print("hello from zpp, {s}!\n", .{self.name});
    }
};

// impl Greeter for Friendly  (vtable + thunks elided here)

fn welcome(who: anytype) void {
    who.greet();
}

pub fn main() !void {
    var f = Friendly{ .name = "world" };
    welcome(&f);
}
```

Two things to notice:

1. The `impl Greeter for Friendly` block emitted the method body
   *inside* `Friendly`'s struct decl as `pub fn greet(self: *@This())`,
   so `who.greet()` resolves through Zig's normal method lookup.
2. `welcome(who: impl Greeter)` lowered to `welcome(who: anytype)`.
   No vtable, no indirect call. Pure monomorphization.

## Step 5 — add a second impl

Open `src/main.zpp` and add a second type:

```zig
const Formal = struct {
    name: []const u8,
};

impl Greeter for Formal {
    fn greet(self) void {
        std.debug.print("Good day, {s}.\n", .{self.name});
    }
}

pub fn main() !void {
    var f = Friendly{ .name = "world" };
    var fr = Formal{ .name = "Ada" };
    welcome(&f);
    welcome(&fr);
}
```

Run:

```sh
zig build run
# hello from zpp, world!
# Good day, Ada.
```

`welcome` is monomorphized twice — once per concrete type. The Zig
compiler emits a separate function for `Friendly` and one for
`Formal`. There's no runtime branching.

## Step 6 — add a runtime contract

Make `welcome` reject empty names:

```zig
fn welcome(who: impl Greeter) void
    requires(@TypeOf(who).* == void or true)  // any boolean expression
{
    who.greet();
}
```

That contract is trivially true. Make a useful one — but for that
you need to read a field. Let's give `Greeter` a `name` accessor:

```zig
trait Greeter {
    fn greet(self) void;
    fn nameLen(self) usize;
}

impl Greeter for Friendly {
    fn greet(self) void {
        std.debug.print("hello from zpp, {s}!\n", .{self.name});
    }
    fn nameLen(self) usize {
        return self.name.len;
    }
}

impl Greeter for Formal {
    fn greet(self) void {
        std.debug.print("Good day, {s}.\n", .{self.name});
    }
    fn nameLen(self) usize {
        return self.name.len;
    }
}

fn welcome(who: impl Greeter) void
    requires(who.nameLen() > 0)
{
    who.greet();
}
```

Run with a valid name:

```sh
zig build run
# hello from zpp, world!
# Good day, Ada.
```

Now make it fail:

```zig
pub fn main() !void {
    var empty = Friendly{ .name = "" };
    welcome(&empty);
}
```

```sh
zig build run
# thread 12345 panic: zpp.contract.requires failed:
#   requires: who.nameLen() > 0 in welcome
```

The panic message tells you which contract failed and inside which
function. In a `ReleaseFast` build the contract compiles out entirely
(via `std.debug.runtime_safety` checks).

## Step 7 — switch to a `dyn` slice

So far each `welcome` call is monomorphized. If you want to mix
multiple concrete types in one collection, you need dynamic dispatch.
Replace the body of `main` with:

```zig
pub fn main() !void {
    var f = Friendly{ .name = "world" };
    var fr = Formal{ .name = "Ada" };

    const chain = [_]dyn Greeter{
        zpp.Dyn(Greeter_VTable).init(Friendly, &f, &Greeter_impl_for_Friendly),
        zpp.Dyn(Greeter_VTable).init(Formal, &fr, &Greeter_impl_for_Formal),
    };

    for (chain) |g| {
        if (g.vtable.nameLen(g.ptr) > 0) {
            g.vtable.greet(g.ptr);
        }
    }
}
```

Run:

```sh
zig build run
# hello from zpp, world!
# Good day, Ada.
```

The `for (chain)` loop now has a runtime dispatch through the
visible `g.vtable.greet` field. Notice that the `dyn` type
appears in the slice literal (`[_]dyn Greeter{ ... }`) — there's
no implicit virtual dispatch happening anywhere.

## Step 8 — split your program across multiple files

So far the project has been a single `src/main.zpp`. Real programs
have more than one file. Add a sibling module:

```sh
cat > src/util.zpp <<'ZPP'
const std = @import("std");

pub fn shout(msg: []const u8) void {
    std.debug.print("UTIL: {s}\n", .{msg});
}
ZPP
```

In `src/main.zpp`, import it by file name:

```zig
const std = @import("std");
const util = @import("util.zpp");

pub fn main() !void {
    util.shout("hello from main");
}
```

The Zig++ compiler rewrites `@import("util.zpp")` to
`@import("util.zig")` during lowering, so the standard Zig import
resolver finds the lowered sibling at compile time.

```sh
zig build run
# UTIL: hello from main
```

The two files are lowered independently and end up next to each
other in `.zpp-cache/` (or wherever your `build.zig` puts them).
There is no implicit "module graph" — you can mix and match `.zpp`
and `.zig` files in the same directory and `@import` either
extension explicitly.

### No `build.zig` needed

`zpp build` works even when your project has no `build.zig`. It
walks every `.zpp` file under the current directory, lowers each
into `.zpp-cache/`, and — if no top-level `build.zig` is present —
writes a minimal `.zpp-cache/build.zig` shim that compiles
`src/main.zig` (or `main.zig`, whichever exists) as the executable
with the `zpp` runtime wired in. Then it invokes
`zig build --build-file .zpp-cache/build.zig`.

```sh
# In a directory with main.zpp + util.zpp and no build.zig:
zpp build
# zpp build: lowered 2 file(s), 0 error(s)
# (zig build runs against the auto-generated shim)
./.zpp-cache/zig-out/bin/<dirname>
```

The shim is regenerated on every run, so editing it by hand is not
useful — drop a real `build.zig` at the project root if you need to
customize. `zpp init` scaffolds one (with `build.zig.zon` and the
`zigpp` dependency declared) for projects you want to publish.

A worked 2-file example lives under `examples/multi_file/` in the
repo. Run it with:

```sh
zig build multi-e2e
# main: hello from util.zpp
# main: this main.zpp ...
# main: imports util.zpp ...
# main: via @import("util.zpp")
```

## Bonus — explaining a diagnostic code

When the compiler prints a code like `Z0020`, you can ask the CLI for
a long-form explanation:

```sh
zpp explain Z0020
```

You get a 1-paragraph summary, an example that triggers the code,
and an example of the fix. Lower-case (`zpp explain z0020`) works
too.
