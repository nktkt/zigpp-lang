# v0.2 — design notes (draft)

This page tracks the major changes being considered for the next
minor version. None of these are committed; they are here to attract
discussion in [GitHub Discussions](https://github.com/nktkt/zigpp-lang/discussions/categories/ideas)
before any code lands.

The doctrine in [MANIFESTO.md](./manifesto.md) still applies. Every
proposal below must justify itself against "no hidden allocation /
control flow / destructor / dispatch / cost".

## ✅ Real async runtime

`zpp.async_mod.TaskGroup` is now a real concurrent executor.

Done:
- `spawn` launches each task on its own OS thread via
  `std.Thread.spawn` and returns a typed `*JoinHandle(T)` that
  publishes the task's result (or error) over a release/acquire
  edge on an atomic state tag.
- `TaskGroup.join()` waits on every worker; on first failure it
  sets the group's `CancellationToken` so cooperative tasks can
  short-circuit, then returns the first error after every thread
  has been reaped.
- `Task` is preserved as `JoinHandle(void)` so callers that
  ignore the spawn result keep compiling unchanged
  (`examples/async_group.zpp` runs end-to-end under `zig build e2e`).
- `tests/behavior/async.zig` covers a 100-task stress matrix
  (concurrent completion, distinct payloads, typed results, single-
  failure error propagation).

Still TODO (tracking, not blocking 0.2):
- Migrate the implementation to Zig 0.17+'s `std.Io` interface
  (`Io.async` / `Io.cancel`) once the API stabilises. The
  `JoinHandle(T)` surface is the seam that lets the swap happen
  without churning callers.
- `.noasync` effect interaction with the new I/O model (Phase 7).

## ✅ `zpp init` template generator

A new subcommand on the main CLI:

```sh
zpp init my-project
```

Creates `my-project/` with `build.zig`, `build.zig.zon`,
`src/main.zpp`, `.gitignore`, and a minimal example. Saves users
from copy-pasting the scaffold.

Uses `@embedFile` on the templates baked into the binary. No
dependency on the source tree.

## ✅ Field-level borrow checking (round 2 landed)

`own var` + `move x` is the flat affine layer. On top of that, sema
tracks `&x` sites and rejects a `move x` whose value still has a live
borrow (Z0021). Borrow recording is *lexical* and per-fn-body — there
is no lifetime annotation system and no cross-fn propagation.

Shape we accept:

```zigpp
own var person = Person{ .name = "Ada", .email = "..." };
const name = &person.name;               // records a borrow on `person`
person.email = "new@example.com";        // OK — disjoint field
print(name);                              // OK — borrow still alive
```

Constraints:
- Borrow is *lexical* (scope-bounded), not lifetime-annotated. The
  full Rust system is out of scope.
- Borrow checker only fires inside a function body. No cross-fn
  inference.
- `move` of a struct invalidates all outstanding live borrows (Z0021).

Done (MVP):
- Purely text-based heuristic. Sema scans raw stmt text in each fn
  body for `&<ident>` / `&<ident>.field` and records borrows per
  name. A `move <ident>` while a borrow is live fires Z0021 ("cannot
  move 'x' while borrowed"). Borrows are scope-local to each fn body
  — the map is reset between fns.
- No new keyword (`let_borrow`) — plain Zig `const r = &x.field;` is
  enough to trigger the check.

Done (round 2):
- **Multi-borrow tracking**: every `&x` site is appended to a
  per-name list (`StringHashMap(ArrayList(BorrowRecord))`) instead of
  the MVP's first-only single span. The Z0021 message now reports
  the offset of the first live borrow so users can find the offending
  `&x`. Multiple borrows of the same name still emit exactly one
  Z0021 per move (regression-tested).
- **Block-scope tracking**: each borrow record carries the lexical
  block depth at which it was recorded. The raw-stmt scanner counts
  `{` / `}` (sharing the existing string / char / `//` skipping
  harness) and on every `}` retires every borrow whose `scope_depth`
  exceeds the new depth. So a `&x` inside `{ ... }` is gone by the
  time control reaches code after the block, and a subsequent
  `move x` no longer trips Z0021.
- The MVP behaviour (borrow at fn-body top level outliving the whole
  body) is preserved because top-level borrows live at depth 0 and
  nothing retires them until end-of-fn.
- Caveat: brace counting from raw text is fragile by construction. We
  re-use `recordBorrows`'s existing string / char / `//` skipping so
  braces inside literals don't perturb the depth, but tricky shapes
  (multi-line `\\` strings, raw string literals when added) may need
  the harness to grow.

Still TODO:
- Cross-fn borrow propagation (callee can move what caller borrowed).
- Distinguishing `&mut` from `&` so a shared borrow doesn't block
  another shared borrow.
- Borrows of expressions like `&vec[0]` (currently only `&ident` and
  `&ident.field`).
- A real lifetime / region inference pass once the heuristic stops
  scaling.

Risk: this is a meaningful complexity addition. May need a separate
opt-in `enable_borrow_check = true` per file before becoming
default-on.

## ✅ `derive` extras: `Clone`, `Default`, `Ord`

Currently we derive `Hash`, `Eq`, `Debug`, `Json`. Add:

- `Clone` — `pub fn clone(self: @This(), allocator: Allocator) !@This()`
  for types containing slices/maps. For POD types it's a memcpy and
  doesn't take an allocator.
- `Default` — `pub fn default() @This()` returning the zero value
  with all-zero fields (matches Zig's `std.mem.zeroes`).
- `Ord` — `pub fn cmp(self: @This(), other: @This()) i32` doing
  field-by-field comparison.

All four already have shapes that compose: `derive(.{ Hash, Eq,
Clone, Default, Ord })`.

## ✅ `derive` extras (round 2): `Iterator`, `Serialize`, `Compare`, `FromStr` (#24)

Additional helpers landed alongside the round-1 set. None change
the lowering contract — each is a comptime namespace whose methods
are spliced into the generated struct.

- `Iterator` — `pub fn iter(self: @This()) FieldIter` yielding the
  struct's field names (`[]const u8`) in declaration order. Useful
  when you want generic logging / reflection without the value side
  of `inline for`.
- `Serialize` — `pub fn serialize(self: @This(), allocator: Allocator) ![]u8`
  producing a `key=value;key=value` text form. Pairs with `FromStr`
  for round-trips on POD-shaped structs; reach for `Json` when
  values can contain `=` or `;`.
- `Compare` — `pub fn lt/le/gt/ge(self, other) bool` plus `min`/`max`,
  all backed by `Ord(T).cmp`. Lets sort/search code spell intent
  without naming `cmp` everywhere.
- `FromStr` — `pub fn fromStr(s: []const u8, allocator: Allocator) !@This()`
  parsing `key=value,key=value` back to `T`. Numeric/bool/enum
  fields parse via the std helpers; `[]const u8` fields are
  duplicated into `allocator` (use an arena to free in one shot).

Composes with the existing set: `derive(.{ Hash, Eq, Ord, Default,
Compare, Iterator, Serialize, FromStr })` is the canonical
"give-me-everything" line.

## ✅ Effect inference (MVP landed; replacing the lint)

Today `effects(.noalloc)` was a heuristic sema lint that flagged any
identifier containing `alloc`/`Allocator`/etc. in raw stmt text. The
MVP replaces it with a bottom-up inference pass that also catches
transitive violations across the local call graph.

Proposal: replace with bottom-up inference.

- Each fn computes its effect set from its body's calls.
- A fn that calls `allocator.alloc` gets `.alloc` automatically.
- An explicit `effects(.noalloc)` annotation becomes a *check* —
  if inference disagrees, emit Z0030.
- The annotation is optional; the inferred set is queryable via
  `@effectsOf(f)`.

Benefit: catches transitive violations (`f` is `.noalloc`, `f`
calls `g` which allocates) without users having to track manually.

Done (MVP):
- `compiler/sema.zig`: `checkEffectInference` runs once per file
  after the per-fn passes. It collects every fn declared in the
  file (top-level, `impl Trait for T`, `owned struct`, `struct`),
  builds a name → index table, scans each body for direct
  allocation (`.alloc(`, `.create(`, `.realloc(`, `.dupe(` —
  string / char / `//` comments stripped), and records any callee
  whose name appears as `<name>(` in the body text. A fixed-point
  loop propagates `.alloc` along callees until stable.
- Z0030 fires when a fn declares `effects(.noalloc)` and the
  inferred set contains `.alloc`. The diagnostic span is the
  offending statement when the local body allocates and the fn
  signature span when the violation is purely transitive (the
  message text says "transitively via a callee" for the latter).
- `tests/diagnostics/diags.zig`: positive (annotation matches
  inference, no Z0030) and negative (transitive violation fires
  exactly once) pairs added alongside the existing direct-call
  Z0030 test.

Done (round 2: `.io` / `.noio`):
- `compiler/sema.zig`: `checkIoEffectInference` runs as a parallel
  pass right after the `.alloc` one. Same shape (collect fns →
  build name → idx → seed direct hits → fixed-point propagate →
  emit Z0030). The "direct IO" heuristic looks for these
  substrings in body text (after stripping strings / chars /
  `//` comments): `std.debug.print`, `std.fs.`, `std.io.`,
  `std.process.`, `std.net.`, `std.os.`, `std.posix.`,
  `try writer.`, `.writeAll(`, `.writeLine(`, `.print(`, `.read(`,
  `.openFile(`, `.createFile(`. Set is intentionally narrow —
  broaden in a follow-up PR.
- `compiler/ast.zig` + `compiler/parser.zig`: `EffectKind` gained
  `.noio`; the parser now recognises `effects(.noio)`.
- `compiler/diagnostics.zig`: Z0030 long-form description covers
  both the `.noalloc` and `.noio` variants.
- `tests/diagnostics/diags.zig`: positive (`.noio` matches
  inference, no Z0030) and negative (`.noio` fn that does IO
  fires Z0030) tests added.

Done (round 3: `.panic` / `.nopanic`):
- `compiler/sema.zig`: `checkPanicEffectInference` runs as a third
  parallel pass right after the `.io` one. Same shape (collect
  fns → build name → idx → seed direct hits → fixed-point
  propagate → emit Z0030). The "direct panic" heuristic looks
  for these substrings in body text (after stripping strings /
  chars / `//` comments): `@panic(`, `std.debug.assert(`,
  `std.debug.panic(`, `std.process.exit(`, plus a NARROWED
  `unreachable` check — the bare keyword must appear at a
  stmt-like position (preceded by start-of-text, `\n`, `;`,
  `{`, `}`, or `=>`) so switch arms `else => unreachable` and
  bare `unreachable;` statements count, but boolean chains like
  `cond or unreachable` do not. Set is intentionally narrow —
  broaden in a follow-up PR.
- `compiler/ast.zig` + `compiler/parser.zig`: `EffectKind` gained
  `.nopanic`; the parser now recognises `effects(.nopanic)`.
- `compiler/diagnostics.zig`: Z0030 long-form description covers
  the `.nopanic` variant alongside `.noalloc` and `.noio`.
- `tests/diagnostics/diags.zig`: positive (`.nopanic` matches
  inference, no Z0030) and negative (`.nopanic` fn that
  `@panic`s fires Z0030) tests added.

Done (round 4: `@effectsOf(f)` queryable surface):
- `compiler/sema.zig`: each of `checkEffectInference`,
  `checkIoEffectInference`, `checkPanicEffectInference` now tees
  its per-fn `inferred[i]` result into a new
  `SemaResult.inferred_effects: StringHashMap(InferredEffects)`
  table (`{alloc, io, panic}` packed bool fields). The existing
  per-pass logic and Z0030 emission are unchanged — the table is
  populated alongside, OR-ing into any existing entry so the three
  passes share the record.
- `compiler/lower_to_zig.zig`: `Lowerer` gained an optional
  `inferred_effects: ?*const InferredEffectsMap` pointer (set via
  the new `initWithEffects` constructor / `lowerWithEffects`
  convenience fn). `writeRewrittenCode` recognises
  `@effectsOf(<ident>)` as a textual rewrite (ahead of the
  existing `@import("...zpp")` branch) and substitutes the
  comma-separated string `"alloc,io,panic"` (omitting absent
  flags; empty string for pure).
- `compiler/mod.zig`: `compileToZig` calls the new
  `lowerWithEffects` and forwards `res.inferred_effects` so the
  end-to-end pipeline sees the substitution.
- `compiler/diagnostics.zig`: new code Z0050
  (`z0050_unknown_fn_in_effects_of`) — fires when
  `@effectsOf(<ident>)` references a name not declared in the
  same `.zpp` file. The lowering still emits `""` so callers
  compile, the diagnostic surfaces the typo / cross-file lookup.
- `tests/diagnostics/diags.zig`: positive `@effectsOf(pure)` →
  `""`, positive `@effectsOf(allocOnly)` → `"alloc"`, and
  negative `@effectsOf(unknown_fn)` → Z0050 tests.
- `examples/effects_of.zpp`: prints the queryable result for two
  contrasting fns at runtime.

Note on the lowering choice: the substitution lives in the
lowerer's textual `writeRewrittenCode` pass (next to
`@import("...zpp")`), not as a parser AST node. `@effectsOf` is a
purely comptime-resolved call site whose target must be looked up
in a sema-built table — synthesising it textually keeps the
parser surface small (no new AST variant, no new token) and
matches the pattern already used by `@import` rewriting.

Still TODO:
- Inference for `.custom` effect kinds.
- Cross-file effect propagation. Today only fns declared in the
  same file are considered when propagating; a `.noalloc` fn that
  calls into an imported allocating fn is silently accepted.
- A real call-graph SCC pass (current heuristic over-approximates
  on indirect / vtable calls and ignores name shadowing across
  impl blocks).
- `@effectsOf` on impl methods (today the same-file table is
  keyed by bare fn name, so methods are looked up by short name
  with no `Type.method` qualifier).

Risk: requires a call graph. Today the compiler is local-stmt-text
based — adding a real call graph is a non-trivial refactor.

## ✅ Multi-file projects (complete)

`@import("other.zpp")` now lowers to `@import("other.zig")` so a
`.zpp` file can reference a sibling `.zpp`. `zpp build` walks the
project, lowers every `.zpp` into a mirrored `.zpp-cache/`, and
generates a `build.zig` shim when the user has not supplied one.

Done:
- Lowerer: rewrite `@import("X.zpp")` → `@import("X.zig")` in all
  raw text (top-level and statement-level). Strings preserved.
- Tests: `tests/multi/import_rewrite.zig` covers the rewrite,
  string-literal preservation, and non-`.zpp` passthrough.
- `zpp build` walks `src/**/*.zpp` (and the project root), lowers
  each into a mirrored `.zpp-cache/.../*.zig`, and writes
  `.zpp-cache/build.zig` referencing the entry point
  (`src/main.zig` or `main.zig`) when no top-level `build.zig`
  exists. End-to-end exercised by `examples/multi_file/`.
- `pub trait` / `pub impl` visibility crosses files. The lowerer
  honors `is_pub` on `trait` (emitting both `pub const X_VTable`
  and `pub const X = zpp.Dyn(X_VTable)`); vtable instances
  (`pub const Foo_impl_for_Bar: Foo_VTable = .{...}`) are always
  emitted `pub`. `examples/multi_file_pub/` showcases an importing
  file constructing dyn pointers to vtable instances declared in
  a sibling `.zpp` module.

## 🔴 `?Trait` — fallible dispatch

Strawman:

```zigpp
fn maybeGreet(who: ?dyn Greeter) void {
    if (who) |g| g.vtable.greet(g.ptr);
}
```

Today the user writes the `?` themselves on `?zpp.Dyn(...)`. The
proposal would make `?dyn Trait` first-class.

Why 🔴: it's syntactic sugar that adds a parser rule but doesn't
introduce capability. Could go either way; raise on the issue
tracker before implementing.

## 🔴 Operator overloading (limited)

Strawman: allow `op +` / `op -` / `op *` / `op /` / `op ==` only
on `struct` types via:

```zigpp
const Vec3 = struct {
    x: f32, y: f32, z: f32,
    op + (a: Vec3, b: Vec3) Vec3 { return .{ .x = a.x+b.x, ... }; }
};
```

Lowers to `pub fn add(a: Vec3, b: Vec3) Vec3 { ... }` plus a
syntactic-sugar pass that rewrites `a + b` to `Vec3.add(a, b)` if
both operands are `Vec3`.

Why 🔴: directly contradicts the doctrine ("no operator
overloading in MVP"). The argument for: numeric types (matrices,
vectors, fixed-point) are awkward without it. The argument
against: it makes call cost invisible.

If this lands, it lands behind an opt-in `pragma allow_op_overload`
at file level.
