# v0.2 — design notes (draft)

This page tracks the major changes being considered for the next
minor version. None of these are committed; they are here to attract
discussion in [GitHub Discussions](https://github.com/nktkt/zigpp-lang/discussions/categories/ideas)
before any code lands.

The doctrine in [MANIFESTO.md](./manifesto.md) still applies. Every
proposal below must justify itself against "no hidden allocation /
control flow / destructor / dispatch / cost".

## Status legend

- 🟢 likely to land
- 🟡 needs design pass
- 🔴 contentious — discuss first

---

## 🟢 Real async runtime (was: scaffold)

Today `zpp.async_mod.TaskGroup` is a serial executor: `spawn`
allocates a closure, queues it, and `join` runs them in order.

Plan:
- Track Zig 0.16's `std.Io` interface and rebuild `TaskGroup` on
  top of it. The MVP serial executor stays as a fallback for
  non-Io targets.
- Introduce `zpp.async_mod.JoinHandle(T)` so `spawn` can return a
  result instead of being void-only.
- Keep `spawn`/`join`/`cancel` as the only primitives. No implicit
  `await` keyword — every join is a function call.

Risk: stdlib churn. We pin Zig versions in CI, so a major API
break in 0.16 means we wait for the dust to settle before merging.

## 🟢 `zpp init` template generator

A new subcommand on the main CLI:

```sh
zpp init my-project
```

Creates `my-project/` with `build.zig`, `build.zig.zon`,
`src/main.zpp`, `.gitignore`, and a minimal example. Saves users
from copy-pasting the scaffold.

Uses `@embedFile` on the templates baked into the binary. No
dependency on the source tree.

## 🟡 Field-level borrow checking

Today `own var` + `move x` is a flat affine system: a name is moved
or it isn't. There is no concept of borrowing a single field.

Proposal:

```zigpp
own var person = Person{ .name = "Ada", .email = "..." };
let_borrow name = &person.name;          // immutable borrow of one field
person.email = "new@example.com";        // OK — disjoint field
print(name);                              // OK — borrow still alive
```

Constraints:
- Borrow is *lexical* (scope-bounded), not lifetime-annotated. The
  full Rust system is out of scope.
- Borrow checker only fires inside a function body. No cross-fn
  inference.
- `move` of a struct invalidates all outstanding borrows (Z0021).

Risk: this is a meaningful complexity addition. May need a separate
opt-in `enable_borrow_check = true` per file before becoming
default-on.

## 🟡 `derive` extras: `Clone`, `Default`, `Ord`

Currently we derive `Hash`, `Eq`, `Debug`, `Json`. Add:

- `Clone` — `pub fn clone(self: @This(), allocator: Allocator) !@This()`
  for types containing slices/maps. For POD types it's a memcpy and
  doesn't take an allocator.
- `Default` — `pub fn default() @This()` returning the zero value
  with all-zero fields (matches Zig's `std.mem.zeroes`).
- `Ord` — `pub fn cmp(self: @This(), other: @This()) i32` doing
  field-by-field comparison.

All four already have shapes that compose: `derive(.{ Hash, Eq,
Clone, Default, Ord })`.

## 🟡 Effect inference (replacing the lint)

Today `effects(.noalloc)` is a heuristic sema lint that flags
identifiers containing `alloc`/`Allocator`/etc. in raw stmt text.

Proposal: replace with bottom-up inference.

- Each fn computes its effect set from its body's calls.
- A fn that calls `allocator.alloc` gets `.alloc` automatically.
- An explicit `effects(.noalloc)` annotation becomes a *check* —
  if inference disagrees, emit Z0030.
- The annotation is optional; the inferred set is queryable via
  `@effectsOf(f)`.

Benefit: catches transitive violations (`f` is `.noalloc`, `f`
calls `g` which allocates) without users having to track manually.

Risk: requires a call graph. Today the compiler is local-stmt-text
based — adding a real call graph is a non-trivial refactor.

## 🟡 Multi-file projects

The current frontend handles one `.zpp` file at a time. A real
project has multiple files. Plan:

- `zpp build` walks `src/**/*.zpp`, lowers each into a mirrored
  `.zpp-cache/src/**/*.zig`, and writes a `build.zig` shim that
  references them.
- Cross-file `trait` references just work because Zig's import
  system carries them through.
- `pub trait` / `pub impl` modifier control visibility across
  files (today `pub` is parsed but only matters for fn/struct).

Caveat: this is "build orchestration" not "compiler" work.
Nothing changes in `compiler/`.

## 🔴 `?Trait` — fallible dispatch

Strawman:

```zigpp
fn maybeGreet(who: ?dyn Greeter) void {
    if (who) |g| g.vtable.greet(g.ptr);
}
```

Today the user writes the `?` themselves on `?zpp.Dyn(...)`. The
proposal would make `?dyn Trait` first-class.

Why 🔴: it's syntactic sugar that adds a parser rule but doesn't
introduce capability. Could go either way; raise on the issue
tracker before implementing.

---

## 🔴 Operator overloading (limited)

Strawman: allow `op +` / `op -` / `op *` / `op /` / `op ==` only
on `struct` types via:

```zigpp
const Vec3 = struct {
    x: f32, y: f32, z: f32,
    op + (a: Vec3, b: Vec3) Vec3 { return .{ .x = a.x+b.x, ... }; }
};
```

Lowers to `pub fn add(a: Vec3, b: Vec3) Vec3 { ... }` plus a
syntactic-sugar pass that rewrites `a + b` to `Vec3.add(a, b)` if
both operands are `Vec3`.

Why 🔴: directly contradicts the doctrine ("no operator
overloading in MVP"). The argument for: numeric types (matrices,
vectors, fixed-point) are awkward without it. The argument
against: it makes call cost invisible.

If this lands, it lands behind an opt-in `pragma allow_op_overload`
at file level.
